# network ‚Äî tests

## How to run
- Fast (Basic + Standard): `./test.sh` (or via `test-module.sh network`; the legacy `firewall` module name is also accepted and ignored as an arg). Takes seconds‚Äìminutes; mirrors all output to `~/logs/firewall-test-<timestamp>.log`.
- Deep: `./test.sh --deep` **or** `TAPPAAS_TEST_DEEP=1 ./test.sh` ‚Äî both enable the deep tier. Expected runtime 5‚Äì10 min (provisions real test VMs, mutates the deployed `zones.json` and OPNsense, then tears down).
- `--no-cleanup` (or `TAPPAAS_TEST_NO_CLEANUP=1`) leaves the test VMs and activated zones in place after a deep run.
- `TAPPAAS_DEBUG=1` adds verbose output.
- Exit codes: 0 pass (or `firewallType=NONE` ‚Üí all skipped); 1 a test failed; 2 fatal (a Basic DNS/connectivity/OPNsense check failed ‚Üí firewall is unreachable, so `update-module.sh` rolls back).
- Prerequisites: sources `common-install-routines.sh` + `cluster/lib/vm-net.sh`; config resolved from `~/config/network.json` (fresh) or `firewall.json` (legacy); OPNsense host fixed at `firewall.mgmt.internal`. Deep tier needs reachable Proxmox nodes + live OPNsense and a working `srvHome`-style trunk.

## Standard (fast) tests
- **Basic 1‚Äì3 (always)** ‚Äî internal + external DNS resolution; ping `1.1.1.1`/`8.8.8.8`; OPNsense reachability (API TCP 443/8443 auto-detect, SSH key-auth login, `opnsense-firewall test` API probe). A failure in this block is FATAL (exit 2). `firewallType=NONE` short-circuits the whole suite to a clean skip.
- **Standard 1: CLI tools available** ‚Äî asserts the plane bins `network-manager` shells out to are on PATH: `opnsense-firewall zone-manager dns-manager caddy-manager rules-manager proxmox-controller switch-controller ap-controller network-manager`. (This is the **plane-bin-resolves** assertion ‚Äî it uses the post-split names, closing the earlier gap where the `switch-manager`‚Üí`switch-controller` rename went undetected.)
- **Standard 1b: ADR-008 provider unit tests** ‚Äî runs `scripts/test-unifi-plugin.sh` (the unifi plugin offline unit suite; this is where the `tagged_vlan_mgmt:custom` regression is covered ‚Äî see notes). The proxmox/switch/ap unit tests moved to the controllers and run via `controller/test.sh`.
- **Standard 2: Schema files parse and validate** ‚Äî `zones.json` (config + module copy) and `aliases.json` parse; `aliases.json` carries `private_ranges` (RFC1918); no duplicate VLAN tags among Active/Mandatory zones.
- **Standard 3: Zone gateway reachability** ‚Äî pings the `.1` gateway of every Active/Mandatory non-mgmt zone (unreachable downgraded to skip).
- **Standard 4: DNS for in-cluster modules** ‚Äî resolves `<vmname>.<zone>.internal` for up to 3 installed modules; `aliasType=network` modules excluded by design (#255).
- **Standard 5: zone-manager summary** ‚Äî `zone-manager --summary` parses + connects.
- **Standard 6: caddy-manager list** ‚Äî `caddy-manager list` completes.
- **Standard 7: rules-manager dry-run** ‚Äî `add-rules <module> --check-mode` (zero errors) + `list-rules` + `list-rules --orphans` against an installed module declaring ingress/egress.
- **Standard 8: rules-manager NONE-mode fallback** ‚Äî `--firewall-type NONE` against a fixture exits 0 without touching OPNsense.
- **Standard 9: Auto-pinhole compile (#177/#173)** ‚Äî synthetic-fixture NONE-mode compile checks: AC-1 pinhole emitted with correct ports (TCP 9091/9092) + UDP protocol propagated; AC-3 pinhole-allowed-from violation warns-and-skips; AC-4 same-zone ‚Üí no pinhole; bonus: zone access-to already covers it ‚Üí no pinhole; missing `pinhole.json` ‚Üí silent no-op.
- **Standard 10: test-network tooling (#225)** ‚Äî `test-network.sh` present/executable/`bash -n` clean/`--help` advertises `--delete`; `test-network-manager` CLI (or python module) exposes create/delete/status; offline rule-model check (test‚Üíinternet allowed, mgmt‚Üítest allowed, RFC1918 blocked, internet-pass sequenced after all blocks).

## Deep tests (live; --deep / TAPPAAS_TEST_DEEP=1)
- **Deep A: switch-controller ‚Äî zone add/change/remove across reconcile phases (#339)** ‚Äî runs in an **isolated temp CONFIG_DIR** (copy of zones.json), hardware-free (`vendor generic` ‚Üí manual plugin), so it never touches live zones or switch config. Exercises the five reconcile phases (update-desired / interrogate / delta / confirm / reconcile): added zones' VLANs enter `desired.json`; access-port nativeVlan tracks its zone; `actual.json` only changes after `confirm`; VLAN change (961‚Üí965) shows drift on trunk + access; removed zone drops out of desired; unknown vendor ‚Üí manual instructions cite the right port/VLAN and `reconcile --apply` returns rc 2 (needs-manual) without falsely confirming.
- **Deep B: ap-controller (ap-manager) ‚Äî SSID tracking + cross-provider uplink validation (#339)** ‚Äî same isolated temp dir: SSID VLAN auto-tracks its zone (965); `delta` reports create-ssid; uplink validation flags a switch port not carrying the SSID VLAN, then clears once the switch trunk carries it; manual AP instructions cite the SSID; `confirm` writes the SSID into `actual.json`.
- **Deep 1‚Äì2: activate test zones + install fixtures** ‚Äî merges fixture test zones (`test-fw-a/b/c`, names derived from fixtures, #306) Active into the **deployed** zones.json (preserving runtime-only/variant zones), runs `zone-manager --execute`, `configctl filter reload`, syncs firewall VM net0 trunks via the shared helper, distributes zones.json to nodes, then installs `test-fw-c` (auto-pinhole provider) and `test-fw-a` (consumer) with one retry on the cloud-init race; waits for DNS. Includes a #306 regression guard that NO fixture zone-name literal is hardcoded in the script.
- **Deep 3‚Äì7: rules + aliases** ‚Äî test-fw-a webserver marker; Caddy reverse-proxy entry present; install test-fw-b; rules-manager applied the module-name peer rule and the module-local alias (`test_admin_ips`) rule; auto-pinhole rule `tappaas-svcdep:test-fw-a:web:test-fw-c:9091` present in OPNsense; FQDN alias `tm_test_fw_a` wired into a rule.
- **Deep 8‚Äì9b: live inter-VM connectivity** ‚Äî test-fw-a‚Üítest-fw-b:9090 permitted (pinhole); test-fw-b‚Üítest-fw-a:8080 permitted (declared bidirectional); auto-pinhole permits test-fw-a‚Üítest-fw-c:9091 cross-zone (AC-2, with alias-table reload + retry); negative check test-fw-a‚Üítest-fw-c:22 BLOCKED.
- **Deep 10: reconcile prunes a removed ingress** ‚Äî removes an ingress entry from deployed test-fw-b.json (Pattern A aware) and asserts `rules-manager reconcile` deletes ‚â•1 orphan rule.
- **Deep 11: Caddy public + split-horizon (ADR-005 #316)** ‚Äî external passthrough via public IP (NAT reflection + wildcard TLS + Caddy‚Üíupstream), internal split-horizon DNS resolves FQDN to the DMZ gateway, and internal reachability through Caddy. Gated/skipped if no public default-environment domain or the public A record doesn't resolve to a public IP.
- Live resources exercised: Proxmox (VM provisioning), OPNsense (zones/VLANs, filter + alias reload, rules, NAT reflection), DNS/DHCP/Unbound, Caddy, and the switch/ap providers (file-only via the manual `generic` plugin ‚Äî NO real switch/AP hardware).

## Coverage notes ‚Äî plane/controller coverage matrix
- **opnsense plane** ‚Äî well covered: `zone-manager`, `rules-manager`, `caddy-manager`, `opnsense-firewall` exercised live in both Standard (summary/list/dry-run/NONE-mode) and Deep (real rules, aliases, NAT reflection, split-horizon, inter-VM connectivity). Strongest coverage of the four planes.
- **switch plane (switch-controller)** ‚Äî covered structurally (full five-phase reconcile lifecycle in Deep A) but **only via the `generic`/manual plugin in an isolated temp dir** ‚Äî NO live switch hardware, no real apply. The vendor-automation (UniFi) apply path is not driven by switch-controller here.
- **ap plane (ap-controller)** ‚Äî same as switch: Deep B covers SSID tracking, uplink validation, and the manual reconcile lifecycle file-only with a `generic` AP ‚Äî NO live AP hardware. (Note: ap-controller is flagged ADR-008 WIP elsewhere.)
- **proxmox plane (proxmox-controller)** ‚Äî **only smoke-tested here**: presence on PATH (Standard 1). Its provider unit tests live in `controller/proxmox-controller` and run via tappaas-cicd's `controller/test.sh`, not in this suite.
- **unifi plugin** ‚Äî offline unit suite `scripts/test-unifi-plugin.sh` runs in Standard 1b; this is where the **`tagged_vlan_mgmt:custom` regression** (UniFi Network 10.x trunk apply must send `tagged_vlan_mgmt:custom`, recent fix) is asserted. The plugin is NOT exercised against a live UniFi controller in this suite.
- **network-manager (orchestrator)** ‚Äî only smoke-tested here (on-PATH in Standard 1; `zone list` read-only smoke lives in the tappaas-cicd suite). The end-to-end network-manager **reconcile that drives all four planes is NOT directly invoked** by network/test.sh ‚Äî the planes are tested individually (opnsense live, switch/ap manual-only, proxmox presence-only).
- Gaps: no live switch/AP hardware path; proxmox-controller apply not driven; network-manager top-level reconcile not driven; Deep 11 and several Standard probes degrade to skip without public DNS / optional bins; Basic-block failure is fatal so later tiers won't run if OPNsense is down.
